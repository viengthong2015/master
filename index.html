1. ບັນຫາທີ່ພົບ ຫຼື ສີ່ທ້າທາຍ (The Challenges)

  • Hard-coded strings: ຄຳສັບ (label) ຖືກກຳນົດຕາຍຕົວໃນ Code.
  • ການ Build ລະບົບຄືນ: ທຸກຄັ້ງທີ່ແປໃໝ່ ຫຼືປ່ຽນແປງ ຄຳສັບ ຕ້ອງ Deploy ລະບົບໃໝ່.
  • ຄວາມຊ້ຳຊ້ອນ: ຄຳສັບດຽວກັນຖືກກຳນົດໄວ້ຫຼາຍບ່ອນໃນ Backend.
  • ຄວາມບໍ່ສອດຄ່ອງ: ຂາດມາດຕະຖານການແປ (Technical Terms) ພາຍໃນທີມ.

ແຜນວາດສະແດງຂັ້ນຕອນການແປແບບດັ້ງເດີມ

2. ຈຸດປະສົງຂອງໂຄງການ (Objectives)

  • ອອກແບບ Database Model ທີ່ຮອງຮັບການຂະຫຍາຍຕົວ.
  • ພັດທະນາ Translation Engine ທີ່ເຮັດວຽກແບບ Runtime.
  • ສ້າງ Admin Interface ໃຫ້ຜູ້ໃຊ້ທົ່ວໄປສາມາດແກ້ໄຂພາສາໄດ້ເອງ.
  • ປະເມີນປະສິດທິພາບປຽບທຽບກັບວິທີການແບບດັ້ງເດີມ.

3. ໂຄງສ້າງລະບົບ (System Architecture)

  • [ ຖານຂໍ້ມູນ ]: ເກັບຄຳສັບ (Key) ແລະ ຄຳແປ (Value).
  • [ ການເຊື່ອມຕໍ່ ]: ຮັບຄຳແປຜ່ານ API ຫຼື/ແລະ Server-Client (java RMI).
  • [ ເຄື່ອງມືການແປ ]: ສະແດງ ຄຳສັບໃສ່ UI ແບບ Dynamic (Runtime).
  • [ UI ]: ປ່ຽນແປງ/ປັບປຸງຄຳແປໂດຍບໍ່ຕ້ອງ Re-build.

ໂຄງສ້າງການເຮັດວຽກຂອງລະບົບແບບ Runtime

4. ການອອກແບບຖານຂໍ້ມູນ (Database Design) ແລະ ເຕັກໂນໂລຊີ (Technology)

  • ທຸກຄຳສັບ (label) ຕ້ອງຈັດເກັບແບບ Single Source of Truth.
  • ຮອງຮັບການ Reuse ຄຳສັບດຽວກັນໃນທຸກ Module.
  • ເກັບຂໍ້ມູນ: Language (ID, Code, Font), UI_Keys, Translations (Value).
  • Technologies: Java-RMI, Java-Swing, Python/Node.js ແລະ Django/react framework.
  • ຮູບແບບ Application: Web Application ແລະ Windows Application

5. ຜົນປະໂຫຍດ ແລະ ການປະເມີນ

  • ຄວາມຢືດຢຸ່ນສູງ (High Agility) ໃນການປັບປ່ຽນ ພາສາໃນ UI
  • ປະຢັດເວລາໃນການແປ, ການພັດພັດທະນາ ແລະ ຫຼຸດຜ່ອນຄ່າໃຊ້ຈ່າຍ
  • ສະດວກຕໍ່ຜູ້ໃຊ້
  • ຜູ້ໃຊ້ສາມາດຈັດການ ການແປດ້ວຍຕົນເອງ
